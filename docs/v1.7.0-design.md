# Smart Memory v1.7.0 - Design Notes

*Working document. Add ideas here as they come up. Nothing here is committed to
until it moves into an implementation plan.*

---

## Theme: Setup quality, configuration clarity, and narrative depth

1.7.0 combines two threads: making Smart Memory easier to get right out of the
box, and deepening the narrative intelligence of what it extracts and injects.
The setup and configuration improvements lower the barrier for new users; the
narrative features give experienced users more powerful tools for complex stories.

---

## Planned features

### Extraction model test button - IMPLEMENTED

A button in the settings panel that sends a fixed sample chat snippet through
the full extraction pipeline and displays the raw output in the panel. No
scoring, no grading - just transparency. The user sees exactly what their
configured model produces for a known input and can judge for themselves whether
it looks reasonable.

**Value:** Helps users pick a good model at setup time and diagnose issues when
extraction goes quiet, without having to commit to a whole session with a bad
model choice.

**Scope:**

- Fixed sample prompt (a short scene with a mix of facts, relationships, and a
  potential arc - enough to exercise all extraction tiers)
- Padded to a realistic context size to surface models that struggle with longer
  prompts
- Output displayed as-is in a scrollable area - no interpretation, no pass/fail
- Button lives in the LLM source section of settings, near the model selector
- Works with all configured sources: main, Ollama, OpenAI-compatible, WebLLM

**Out of scope:** Content tolerance testing (would require distributing
sensitive fixtures in the repo). Scoring or automated grading (requires a judge
model at least as capable as the one being tested, which defeats the purpose for
local models).

### Pinned arc transfers for group chats - IMPLEMENTED (including resolved state for pinned arcs)

Pinned arcs currently only carry over to new chats in solo sessions - in group
chats the pin button is hidden entirely because there is no single character to
tie the arc to. This is a known limitation that affects users running long
multi-session group roleplays.

**Approach:** Store pinned arcs against the group ID rather than a character
name, keyed in `extension_settings` the same way long-term memories are keyed
by character name. Group IDs in SillyTavern are stable `Date.now()` strings
generated at group creation - they do not change after creation and only become
orphaned if the group is deleted, which is acceptable.

**Value:** Brings group chats to parity with solo chats for arc continuity.
Users running long multi-session group stories currently have no way to carry
open threads forward automatically.

**Scope:**

- Show the pin button in group chats (currently hidden when `groupId` is set)
- On `/newchat`, load pinned arcs from the group ID store and seed the new chat
- Arc storage under `extension_settings.smart_memory.group_arcs[groupId]`
- Cleanup: orphaned group arc stores can be left in place - they are small and
  inert once the group is gone

### Perspective-scoped knowledge ("who knows what?")

Track knowledge per character: who witnessed an event, who was told later, who
is lying, who suspects something, who still does not know a secret. Inject only
the knowledge appropriate to the responding character.

Examples:

- "Lyria knows Kael stole the relic; Kael does not know she knows."
- "The player knows the assassin is Mira, but Arlen still believes it was a bandit."
- "Nessa is pretending to forgive Corvin, but privately plans to betray him."

**Value:** Solves accidental omniscience - one of the most common immersion
breakers in AI roleplay. Characters react to information they were never present
for because the memory system has no concept of whose knowledge is whose. The
entity registry already tracks characters and relationships; epistemic scoping
adds a missing semantic layer over those entities.

**Scope:**

- New extraction pass for epistemic facts: who knows X, who suspects X, who is
  deceiving whom about X
- Stored per character alongside their long-term memories
- Injected as a "private knowledge" block for the responding character, and
  optionally a "things {{char}} does not know" block to actively suppress leakage
- Most valuable in group chats and intrigue/mystery/betrayal stories

**Tradeoffs:** Requires a capable model to extract reliably - epistemic states
are subtle and easy to get wrong from ambiguous prose. Likely Profile B only.
One bad knowledge assignment can cause persistent dramatic errors, so a review
UI similar to the continuity checker may be needed.

### Structured state ledger (entity registry enhancement)

Extend the entity registry to track mutable current state for important
entities, separate from prose memories. Rather than a new system, this evolves
what the entity registry already does into something more precise for high-churn
fields:

- Character: current location, injuries, disguise, active goal, carried items
- Object: current owner, location, condition, whether lost/destroyed
- Place: current occupants, hazards, accessibility, who controls it
- Faction: current leadership, objective, alliances, hostility level

**Value:** Continuity often fails on physical logistics - where the knife went,
who has the key, whether the door is locked, whether someone is still injured.
Memories record these as facts but a structured current-state slot makes mutable
"where/status/owner" details easier to update and retrieve without digging
through the memory list. Supersession already retires outdated memories when
facts change; this makes that concept explicit for the most commonly-updated
entity properties.

**Scope:** To be designed. Likely an opt-in extension of the entity registry
panel with editable structured fields per entity type, extracted and updated
during the same extraction pass that currently populates the registry.

**Tradeoffs:** State extraction is harder than memory extraction - the model
must distinguish current state from past state, and conflicts need clear
resolution rules. Scoped as optional to avoid complexity for users who do not
need it.

### Relationship state with deltas

The `relationship` memory type currently captures that a relationship exists
("Elara is Ren's sister") but not its emotional temperature. Models lose track
of the *state* of relationships fastest in long RP - trust, affection, tension,
debt, suspicion - which is what actually drives in-character reactions.

**Approach:** Per-pair state object updated each scene as deltas rather than
absolutes, injected as a single compact line per relationship.

Example injection format:

```text
Elara→Ren: warm, cautious, owes a favor
Lyria→Kael: hostile, suspicious, recently betrayed
```

**Value:** Tightens characterization without bloating the prompt. A character's
default reaction to another character should reflect the current state of their
relationship, not just the historical fact of it. This is one of the most
common ways AI roleplay drifts: the relationship was rocky three scenes ago and
the model has forgotten because nothing in the injected memory says so now.

**Scope:**

- Per-character-pair state stored alongside long-term memories
- Compact descriptors (warm/cool/hostile, trusting/suspicious, free/indebted, etc.)
- Updated during the same extraction pass that handles relationship facts
- Injection appends one line per active relationship pair to the long-term block

**Tradeoffs:** Needs a capable extractor to infer deltas reliably. Weak local
models will produce noisy state changes - a confidence threshold or manual
smoothing may be needed. Likely Profile B works best; Profile A may need a
simplified version with fewer state dimensions.

### Witnessed-by tagging for group chat knowledge filtering

Tag each memory at extraction time with the set of characters present in the
scene it was extracted from. On injection for a group member, filter out
memories the responding character did not witness.

**Approach:** Use the existing scene tracking to derive a `witnessed_by` set
per memory. Inject only memories where the responding character is in that set,
or downgrade non-witnessed memories to a separate "rumor / heard about" block
with appropriate framing.

**Value:** Addresses the known limitation where newly-joined group members
inherit shared context for events they were never present for. Membership is
per-memory rather than a contiguous range, so characters joining and leaving
multiple times across a chat works correctly.

**Scope:**

- New `witnessed_by` field on memory entries
- Populated during extraction from current scene membership
- Injection filter applied per group member based on responding character
- Optional "secondhand knowledge" framing for memories outside the witness set

**Tradeoffs:** Requires reliable scene-membership detection, which is heuristic
today. False negatives leak knowledge to characters who should not have it;
false positives hide knowledge they should have. Pairs naturally with
perspective-scoped knowledge above - witnessed-by handles "who was there",
perspective-scoped handles "who was told later" - and the two could share
infrastructure. Worth considering whether they should be designed together as
a single epistemic-tracking subsystem rather than separate features.

### Activation triggers on memories - IMPLEMENTED

The hybrid scorer already does entity overlap with the current turn, but some
memories are activated by topics or conditions rather than entity names. "Elara
is allergic to bee stings" only matters when bees come up. "The locked tower
can only be opened at midnight" only matters when the tower is approached. These
rare-but-critical facts currently either crowd out the budget at high importance
or get evicted at low importance.

**Approach:** A `triggers` array of keyword/entity strings on each memory, with
a score bonus applied in the hybrid scorer when any trigger matches the recent
context window. Complements entity overlap rather than replacing it - captures
conditional relevance rather than co-mention.

**Scope:**

- New `triggers` field on memory schema, added via graph migration
- Extraction prompt updated to suggest triggers alongside each new memory
- Score bonus in `hybridScore` when any trigger matches the rolling context window
- Alternatively: derive triggers from content nouns at write time (cheaper,
  noisier - worth evaluating against LLM-suggested triggers)

**Tradeoffs:** LLM-suggested triggers add prompt complexity to the extraction
call. Noun-derived triggers are cheaper but will miss conditional phrasing.
Probably offer both paths and let the configured model quality determine which
is used.

### Source-message provenance (complete the half-done feature) - IMPLEMENTED

`source_messages: []` is already defined on every memory in `graph-migration.js`
but nothing in the extraction path populates it - the field stays empty across
all existing memories. The schema is there; it just needs to be filled.

**Approach:** During extraction, attach the message-index window the extraction
call ran on to each new memory written in that batch. Add a "jump to source"
button next to the existing edit/delete actions in the memory panel so users can
verify where a memory came from.

**Value:** Users opening the panel months into a long RP can verify why a memory
exists and catch anything that was extracted incorrectly without manually
searching the chat history.

**Scope:**

- Populate `source_messages` with `[startIndex, endIndex]` during
  `extractAndStoreMemories`
- "Jump to source" button in the memory list that scrolls the chat to the source
  range and briefly highlights it
- No schema migration needed - field already exists on all memories

**Tradeoffs:** Message indices shift if messages are deleted (same limitation as
read-only mode ghost ranges). The jump will land in approximately the right place
but may be off by the number of deleted messages. Acceptable - it is a
navigation aid, not a precise link.
